# Технический план для текущей версии HTML Puzzle

## Подход

Этот план продолжает [ROADMAP.md](/d:/other/pazzles/ROADMAP.md) и переводит ближайшие задачи в технические шаги:

- что именно менять;
- в каких файлах это делать;
- какой результат считать готовым.

Порядок этапов выстроен так, чтобы сначала стабилизировать базовую механику пазла, а уже потом развивать прогрессию и дополнительные режимы.

## Этап 1. Стабилизировать геометрию пазла и текстуру изображения

### Цель

Добиться точного совпадения:

- подсказки;
- целевого контура пазла;
- текстуры внутри деталей;
- итоговой собранной картинки.

### Что проверить и поправить

- единая система координат для `board`, `puzzleRect`, `pieceSize` и текстуры;
- корректный `offset`/`scale` для изображения в `headbreaker`;
- влияние `borderFill`, `puzzleDiameter` и формы `Rounded` на видимый сдвиг;
- корректность позиционирования `targetPosition` для каждой детали;
- поведение в режимах `stretch` и `contain`.

### Файлы

- [layout.js](/d:/other/pazzles/src/js/infrastructure/layout/layout.js)
- [puzzle-runtime.js](/d:/other/pazzles/src/js/infrastructure/puzzle/puzzle-runtime.js)
- [puzzle-factory.js](/d:/other/pazzles/src/js/infrastructure/puzzle/puzzle-factory.js)
- [game.js](/d:/other/pazzles/src/js/application/game/game.js)
- [styles.scss](/d:/other/pazzles/src/scss/styles.scss)

### Технические шаги

1. Добавить временную диагностическую конфигурацию с простым контуром детали.
2. Сравнить поведение `Classic/Squared` и `Rounded`.
3. Зафиксировать, где именно появляется смещение:
   в расчёте пазла, в offset текстуры или в форме детали.
4. Упростить модель текстуры до одного предсказуемого сценария.
5. Только после этого вернуть финальный контур деталей.

### Критерий готовности

- верхняя часть изображения в собранном пазле совпадает с подсказкой;
- нет пустых блоков;
- нет вертикального или горизонтального ухода картинки;
- режим `Упорядочить` не вскрывает скрытые артефакты геометрии.

## Этап 2. Переделать механику кнопки `Упорядочить`

### Цель

Сделать кнопку полезной, но не превращающей партию в автоматическую победу.

### Что изменить

- убрать прямое перемещение деталей в `targetPosition`;
- раскладывать детали по зонам, строкам, колонкам или краям поля;
- исключить состояние, в котором кнопка сама даёт `win`.

### Файлы

- [game.js](/d:/other/pazzles/src/js/application/game/game.js)
- [gameplay-stats.js](/d:/other/pazzles/src/js/application/game/gameplay-stats.js)
- [app-copy.js](/d:/other/pazzles/src/js/content/app-copy.js)

### Технические шаги

1. Определить один сценарий упорядочивания:
   например, раскладка по сетке вокруг целевой области.
2. Вынести расчёт позиций упорядочивания в отдельную функцию.
3. Проверить, что после `order` детали не соединяются автоматически.
4. Обновить текст подсказки и штраф режима, если нужно.

### Критерий готовности

- после `Упорядочить` пазл остаётся нерешённым;
- кнопка помогает игроку, но не решает задачу за него;
- статистика и счёт продолжают работать корректно.

## Этап 3. Автосохранение незавершённой партии

### Цель

Позволить игроку закрыть приложение и продолжить пазл с того же места.

### Что хранить

- id изображения из пула;
- режим партии;
- размеры сетки;
- координаты деталей;
- существующие соединения;
- время партии;
- состояние подсказки.

### Файлы

- [game.js](/d:/other/pazzles/src/js/application/game/game.js)
- [image-pool.js](/d:/other/pazzles/src/js/application/game/image-pool.js)
- [timer.js](/d:/other/pazzles/src/js/shared/time/timer.js)
- новый модуль состояния партии:
  `src/js/application/game/session-state.js`
- [main.js](/d:/other/pazzles/src/js/app/main.js)

### Технические шаги

1. Ввести сериализацию текущего состояния пазла.
2. Сохранять состояние по таймеру и при важных действиях игрока.
3. Восстанавливать состояние при запуске приложения.
4. Добавить на стартовом экране выбор:
   продолжить последнюю партию или начать новую.

### Критерий готовности

- после перезапуска приложение предлагает продолжение;
- пазл восстанавливается без перескока изображения и логики режима;
- сохранение не ломает обычный старт новой партии.

## Этап 4. Довести режимы партии до законченной модели

### Цель

Сделать режимы не только косметическими, а действительно различающимися по правилам.

### Что добавить

- прозрачные ограничения и бонусы для каждого режима;
- отображение правил режима до старта партии;
- единый объект режима, который управляет подсказками, таймером, штрафами и текстами.

### Файлы

- [game-modes.js](/d:/other/pazzles/src/js/domain/game/game-modes.js)
- [settings.js](/d:/other/pazzles/src/js/application/settings/settings.js)
- [game.js](/d:/other/pazzles/src/js/application/game/game.js)
- [gameplay-stats.js](/d:/other/pazzles/src/js/application/game/gameplay-stats.js)
- [index.html](/d:/other/pazzles/index.html)
- [app-copy.js](/d:/other/pazzles/src/js/content/app-copy.js)

### Технические шаги

1. Нормализовать контракт режима:
   `hintsAllowed`, `timeLimitMs`, `scoreModifiers`, `uiDescription`.
2. Привязать один источник правды к HUD и логике партии.
3. Добавить предматчевое описание режима в меню/настройках.
4. Проверить граничные сценарии:
   победа в последний момент, запрет подсказки, выход по времени.

### Критерий готовности

- правила режима понятны до старта;
- логика режима не расходится с интерфейсом;
- все режимы проходят одинаково предсказуемый цикл партии.

## Этап 5. Сделать полноценный марафон по пулу изображений

### Цель

Превратить пул изображений в последовательный игровой режим.

### Что добавить

- счётчик прогресса по пулу;
- переход между картинками как часть серии;
- итоговый экран серии;
- отдельное сохранение результата марафона.

### Файлы

- [image-pool.js](/d:/other/pazzles/src/js/application/game/image-pool.js)
- [game.js](/d:/other/pazzles/src/js/application/game/game.js)
- [results.js](/d:/other/pazzles/src/js/application/results/results.js)
- [leaders.js](/d:/other/pazzles/src/js/presentation/leaders.js)
- [index.html](/d:/other/pazzles/index.html)
- [dom.js](/d:/other/pazzles/src/js/presentation/dom.js)

### Технические шаги

1. Ввести сущность серии:
   список изображений, текущий индекс, общий счёт, общее время.
2. Разделить одиночную победу и завершение всей серии.
3. Показать прогресс:
   например `2 из 7`.
4. Сохранить итог серии отдельно от обычных партий.

### Критерий готовности

- игрок видит, сколько изображений осталось;
- завершение всей серии ощущается как отдельное событие;
- таблица результатов не смешивает серию и одиночную игру.

## Этап 6. Коллекция завершённых пазлов

### Цель

Сделать локальную историю прохождения и галерею собранных изображений.

### Что хранить

- имя файла;
- миниатюру;
- лучшее время;
- лучший счёт;
- количество прохождений;
- дату первого и последнего завершения.

### Файлы

- [results.js](/d:/other/pazzles/src/js/application/results/results.js)
- модуль хранения коллекции:
  `src/js/application/collection/puzzle-collection.js`
- [index.html](/d:/other/pazzles/index.html)
- [dom.js](/d:/other/pazzles/src/js/presentation/dom.js)
- [styles.scss](/d:/other/pazzles/src/scss/styles.scss)

### Технические шаги

1. Выделить отдельное хранилище коллекции.
2. На победе обновлять запись пазла.
3. Добавить страницу коллекции с карточками и миниатюрами.
4. Сделать сортировку:
   по дате, по счёту, по времени, по количеству прохождений.

### Критерий готовности

- у игрока появляется видимая история прогресса;
- каждая картинка сохраняет локальную статистику;
- интерфейс коллекции работает без дублирования данных из лидеров.

### Что улучшить

- единый сценарий выпуска.

### Файлы

### Технические шаги

1. Разделить обычную сборку, проверочную сборку и portable.
2. Улучшить сообщения об ошибках очистки `dist`.
3. Добавить отдельную команду для полного релизного цикла.
4. При необходимости перед сборкой завершать зависшие процессы приложения.

### Критерий готовности

- сборка предсказуемо создаёт нужный артефакт;
